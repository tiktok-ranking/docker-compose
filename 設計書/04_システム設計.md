# 04_システム設計書 (System Design)

## 1. 概要
- **対象要件定義書**: `01_要件定義.md`
- **システムの目的**: TikTokライバー実績の統合管理、自動集計パイプライン、およびファン向けエンゲージメント機能の提供。
- **スコープ**: Docker Composeによるオーケストレーション、認証基盤(Keycloak)、データ加工(Python)、フロントエンド(Next.js)。

## 2. 全体アーキテクチャ

### 2.1 構成図 (Container Diagram)
```text
[User Browser]
      | (HTTPS/HTTP)
      v
[Nginx (Reverse Proxy)]
      |
      +-- [/] -----------> [frontend-corp (Next.js)] <---> [Tikfinity API]
      |
      +-- [/ranking] ----> [frontend-rank (Next.js)]
      |                        | (OIDC/PKCE)
      |                        v
      |                  [Keycloak (Auth)] <------------> [PostgreSQL (Keycloak DB)]
      |                        |
      |                        +-- (Verification) ------> [Google Sheets API]
      |
      +-- [Shared Volume] <------------------------------ [worker-fetcher (Python)]
            (CSV/XLSX)                                         | (Sync)
                                                               v
                                                       [Google Sheets API]
```

### 2.2 コンポーネント一覧
| コンポーネント | 役割 | 実行環境 |
| :--- | :--- | :--- |
| **nginx** | パスベースルーティング、SSL終端、Gzip圧縮 | nginx:alpine |
| **frontend-corp** | 自社HP、Tikfinity連携(Next.js, framer-motion, lucide-react) | Node.js 22 (Next.js 15) |

| **worker-fetcher** | Excel加工(Python, Pandas, Selenium, openpyxl) | Python 3.13-slim |
| **shared-data** | Excel原本(raw)、加工済みCSV、ログの永続化領域 | Docker Volume (Bind Mount) |

## 4. コンポーネント設計

### 4.1 認証基盤 (Keycloak & Auth API)
- **役割**: 認証の集中管理。
- **内部処理**:
    - **サインアップガード**: `frontend-rank` 内に `/api/auth/check-whitelist` を実装。
    - スプレッドシート `Livers` シートの `status` が `approved` 且つ `registered` でない場合のみ登録を許可。
- **依存**: `GOOGLE_SERVICE_ACCOUNT_JSON`

### 4.2 データ加工 (worker-fetcher)
- **役割**: データのクレンジングと統合。
- **入出力**:
    - Input: `/app/data/raw/*.xlsx`
    - Output: `/app/data/ranking_data.csv`, Google Sheets `History` シート
- **処理概要**: Pandas DataFrame を使用し、TikTok ID をキーにマスタ（マネージャー名等）を結合。
- **ブラウザプロファイル**: `/app/profile` をボリューム化し Selenium のログインセッションを維持。

### 4.3 ボリューム詳細設計
| ホストパス | コンテナパス | 関連 | 用途 |
| :--- | :--- | :--- | :--- |
| `./data/creators` | `/app/data` | Fetcher, Rank | 抽出済みCSV/Excelの格納・参照 |
| `./data-fetcher/profile` | `/app/profile` | Fetcher | Selenium ブラウザプロファイル（ログイン維持） |
| `./logs` | `/app/logs` | 全体 | 実行ログの集約 |
| `./nginx/conf.d` | `/etc/nginx/conf.d` | Nginx | パスベースルーティング設定 |

## 5. API設計

### 5.1 ガチャ実行 API (Internal Server Action)
- **エンドポイント**: `POST /api/gacha/execute`
- **バリデーション**:
    - `userId`: 必須、Keycloak UUID形式
    - `timestamp`: 必須
- **レスポンス**:
    ```json
    {
      "success": true,
      "prizeId": "p_001",
      "animation": "sparkle_gold",
      "newBalance": 450
    }
    ```

### 5.2 ホワイトリスト照合 API
- **エンドポイント**: `GET /api/auth/check?username=xxxx`
- **エラーレスポンス**:
    - `403 Forbidden`: `{ "error": "NOT_IN_WHITELIST" }`
    - `409 Conflict`: `{ "error": "ALREADY_REGISTERED" }`

## 6. データ設計

### 6.1 エンティティ一覧 (Google Sheets / CSV)

- **格納パス**: `./data/creators/ranking_data.csv`

## 7. 処理フロー

### 7.1 サインアップ・認証フロー (正常系)
1. **ユーザー**: `frontend-rank` のサインアップ画面へアクセス。
2. **ユーザー**: 管理者から通知された「ユーザー名」を入力。
3. **frontend-rank**: `/api/auth/check` を実行し、Google Sheets のホワイトリストを照合。
4. **frontend-rank**: 照合OKの場合、KeycloakのサインアップURLへリダイレクト。
5. **Keycloak**: ユーザーがパスワード等を設定し登録完了。
6. **認証完了**: OIDC Code Flow によりセッション確立。

### 7.2 データ加工フロー (異常系)
1. **worker-fetcher**: `/data/raw` に配置された `.xlsx` を検知。
2. **worker-fetcher**: ファイル読み込み時にカラム名不一致を検知。
3. **エラー処理**: 処理を中断し、対象ファイルを `/data/error/` へ移動。
4. **通知**: Discord Webhook 経由で「データ形式エラー：[ファイル名]」を管理者に送信。

## 8. 非機能設計

### 8.1 パフォーマンス
- **目標**: ランキング表示のLCP (Largest Contentful Paint) を 2.0s 以内とする。
- **対策**: 共有ボリュームのCSVをISR (1時間間隔) でパースし、キャッシュ化。ランキングはTOP 50まで表示。

### 8.2 セキュリティ
- **認可**: ライバー個人の詳細データ表示ページでは、JWTの `sub` (Keycloak ID) とスプレッドシートの `tiktok_id` の紐付けチェックを必須とする。
- **データ保護**: 共有ボリュームへのアクセス権限は、Dockerコンテナユーザーに限定する。

## 9. エラーハンドリング設計

| エラー種別 | 判定基準 | 対応方針 | 影響範囲 |
| :--- | :--- | :--- | :--- |
| 認証失敗 | ホワイトリスト不在 | 専用エラー画面を表示し、管理者に連絡を促す | サインアップ不可 |
| API制限 | Google Sheets 429 | 指数バックオフによるリトライ (最大3回) | データ同期の遅延 |
| データ不備 | Excel カラム欠落 | 処理をスキップし Discord 通知 | 最新データの反映停止 |
| 連携失敗 | Tikfinity API ダウン | ガチャボタンを非活性化 | ガチャ機能の一時停止 |

## 10. 外部連携

### 10.1 Google Sheets API
- **用途**: マスタ参照 (Read)、実績蓄積 (Append)。
- **失敗時の挙動**: 読み取り失敗時はローカルにキャッシュされた前回のCSVを使用して表示を維持する。
- **構成**: `Livers`（マスタ）, `History`（実績）, `Config`（ガチャ景品設定）の3シート構成。

### 10.2 Tikfinity API
- **用途**: ポイント消費、ユーザー残高確認。
- **失敗時の挙動**: トランザクション制御。ポイント減算APIが成功し、かつ抽選処理が成功した場合のみ確定。
- **エンドポイント**: `GET /api/points/${userId}`, `POST /api/points/consume`

## 11. 制約・前提条件

### 11.1 技術的制約
- **Docker Desktop on Windows**: パス指定に `e:\Users\...` 等のホスト絶対パスを Bind Mount で使用。パーミッション問題回避のため、Linux 互換モード (WSL2) を前提とする。
- **DBレス**: 複雑なリレーションクエリは行わず、Pandas またはメモリ上での Join に限定する。

### 11.2 運用上の前提
- **手動エクスポート**: 管理者は TikTok Live Backstage から「過去7日間」の集計データを手動で抽出する必要がある。

## 12. リスクと対策

### 12.1 TikTok UI 変更リスク
- **事象**: TikTok側のExcelエクスポート形式が変更され、加工ロジックが壊れる。
- **対策**: `worker-fetcher` にスキーマ定義レイヤーを設け、カラム名のエイリアス設定のみで追従可能にする。

### 12.2 データ完全性リスク
- **事象**: スプレッドシートを管理者が直接編集し、データ型が壊れる。
- **対策**: `worker-fetcher` の書き込み前にデータバリデーション（型チェック、空文字チェック）を強制する。
