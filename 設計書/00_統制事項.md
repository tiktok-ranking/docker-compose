# 00_統制事項 (Control Matters)

## 1. 目的
本ドキュメントは、TikTokライバーマネジメントシステム全体の品質、セキュリティ、および運用の安定性を担保するための統制事項を定義する。メインリポジトリは各サブプロジェクトのハブとして機能するため、全体整合性の維持を主眼に置く。

---

## 2. コーディング・構成規約
### 2.1 Docker Compose 記述規約
- **目的**: 実行環境の一貫性と保守性の確保。
- **ルール**:
  - **必須**: イメージタグに `latest` を使用せず、特定のバージョンを指定する。
  - **必須**: コンテナ間の依存関係は `depends_on` (condition: service_healthy) を使用して明示する。
  - **必須**: 環境変数は直接 `compose.yaml` に記述せず、`secrets/` 内の `.env` ファイルを参照する。
  - **推奨**: ボリュームマウントは絶対パスを避け、リポジトリルートからの相対パスを使用する。
- **理由**: `latest` の使用は予期せぬアップデートによる動作不良を招き、依存関係の不備は起動順序エラーを引き起こすため。
- **違反時のリスク**: 環境再現性の喪失、OS再起動時の自動復旧失敗。
- **チェック方法**: `docker compose config` による構文チェック、およびコードレビュー。

### 2.2 サブプロジェクト（Submodule）管理
- **目的**: 各サービスの独立性と結合の整合性維持。
- **ルール**:
  - **必須**: サブプロジェクトの `main` ブランチが更新された際、メインリポジトリ側のハッシュ更新を PR (Pull Request) として提出する。
  - **必須**: 独自のパッチをメインリポジトリ側で直接サブプロジェクト内のファイルに当てない（各サブプロジェクトのリポジトリで修正する）。
- **理由**: メインリポジトリとサブプロジェクトの不整合を防ぐため。
- **チェック方法**: PRにおける `.gitmodules` および submodule hash の差分確認。

---

## 3. テスト方針
### 3.1 カバレッジ目標
- **目的**: コードの信頼性とリファクタリング耐性の確保。
- **ルール**:
  - **必須**: 全サブプロジェクトにおいて **C0（命令網羅）80% 以上** を維持する。
  - **推奨**: 重要なビジネスロジック（データ加工、ポイント消費）については C1（分岐網羅）100% を目指す。
- **理由**: 未テストのロジックによるデータ不整合（ギフト数の計算誤り等）を防ぐため。
- **違反時のリスク**: バグの潜在、将来的な仕様変更時のデグレード。
- **チェック方法**: 各コンテナの CI 実行時にカバレッジレポートを出力し、レビュー時に確認する。

---

## 4. セキュリティ（認証・認可・データ保護）
### 4.1 認証基盤 (Keycloak)
- **目的**: 高度なユーザー管理とシングルサインオンの実現。
- **ルール**:
  - **必須**: 全てのフロントエンド（コーポレート/ランキング）および API へのアクセスは、Keycloak による OIDC 認証を経由させる。
  - **必須**: 管理者パスワードは Keycloak のベストプラクティスに基づき、複雑性（英大小数記号）と長さを強制する。
- **理由**: 規約第10条（秘密保持）に基づき、ライバーの機密データを保護するため。
- **違反時のリスク**: ライバーデータの流出、TikTok Live Backstage 権限の悪用。
- **チェック方法**: 認証なしでのエンドポイントアクセス拒否をテストで確認。

### 4.2 秘密情報の管理
- **目的**: APIキーやパスワードの漏洩防止。
- **ルール**:
  - **禁止**: `secrets/` フォルダ内の実ファイルを Git にコミットすることを厳禁とする。
  - **必須**: `.gitignore` に秘密情報ファイルが登録されていることを確認する。
- **理由**: ソースコード共有時の重大なセキュリティ事故を防ぐため。
- **違反時のリスク**: 外部API（Tikfinity）の不正利用、クラウド破産、信頼喪失。
- **チェック方法**: `trufflehog` 等のツールによるシークレットスキャン。

---

## 5. ログ・監視
### 5.1 ログ出力
- **目的**: 障害発生時の早期原因特定。
- **ルール**:
  - **必須**: コンテナのログは `stdout` / `stderr` に出力し、Docker のロギングドライバーが回収できるようにする。
  - **必須**: 機密情報（パスワード、個人収益額）の生データはログに出力しない（各サブプロジェクトでマスク処理を行う）。
- **理由**: ログファイルによるストレージ圧迫回避と、ログからの情報漏洩防止のため。
- **チェック方法**: 開発・テスト時のログ出力サンプル確認。

---

## 6. Git運用（ブランチ戦略・レビュー）
### 6.1 GitHub Flow
- **目的**: 開発スピードの維持と品質の担保。
- **ルール**:
  - **必須**: `main` ブランチへの直接 push は禁止。必ず Feature ブランチから Pull Request を作成する。
  - **必須**: PR のマージには、最低 1 名の承認（Approve）と、自動テストのパスを必須とする。
  - **推奨**: PR の粒度は「1つの機能」または「1つの修正」に限定し、巨大な PR を避ける。
- **理由**: 誤ったコードの混入を最小限に抑え、変更履歴を追跡可能にするため。
- **違反時のリスク**: `main` ブランチの破壊（システム停止）、デバッグ困難なコードの混入。
- **チェック方法**: GitHub の Branch Protection Rules 設定による強制。

---

## 7. デプロイ・リリース管理
### 7.1 手動デプロイ手順
- **目的**: 本番環境更新時のミス防止。
- **ルール**:
  - **必須**: デプロイ作業前には必ず `secrets/` のバックアップを確認する。
  - **必須**: `git pull origin main` 後の `docker compose up -d` 実行時、ログを監視して起動エラーがないか確認する。
  - **推奨**: デプロイ作業ログ（日時、担当者、コミットハッシュ）を残す。
- **理由**: 自動化されていない環境での「属人的な作業ミス」を最小限にするため。
- **違反時のリスク**: デプロイ失敗による長時間ダウンタイム、不完全な状態でのサービス提供。
- **チェック方法**: デプロイチェックリストの運用。

---

## 8. 法令・利用規約遵守
### 8.1 TikTok Live Backstage 規約準拠
- **目的**: アカウント凍結リスクの回避。
- **ルール**:
  - **必須**: `data-fetcher` による TikTok への直接的な自動ログイン・自動エクスポートは行わない（手動エクスポートファイルを起点とする）。
  - **必須**: 取得したデータは、業務委託等の正当な権限を持つ範囲内でのみ利用する。
- **理由**: 規約第7条（自動化スクリプトの禁止）への違反を避けるため。
- **違反時のリスク**: TikTok アカウントの永久停止、法的措置。
- **チェック方法**: 運用フロー（手動作業の介在）の確認、および利用規約の定期的な再確認。

---

## 9. パフォーマンス・負荷対策
### 9.1 コンテナリソース制限
- **目的**: 特定プロセスの暴走による VPS 全体のハングアップ防止。
- **ルール**:
  - **推奨**: `compose.yaml` にて `deploy.resources.limits` を設定し、各コンテナの CPU/メモリ上限を定義する。
- **理由**: Python (Pandas) 等のメモリ消費が激しい処理が他のサービス（Keycloak等）を圧迫するのを防ぐため。
- **チェック方法**: 負荷試験時の `docker stats` 確認。

## 備考（仮定事項）
1. **Keycloak導入**: 本プロジェクトでは認証基盤として Keycloak を Docker Compose 内で構成することを前提としています。
2. **Windows環境**: デプロイ先が ConoHa VPS (Windows) であるため、バッシュスクリプト等は WSL2 または Docker Desktop 上での動作を想定しています。
3. **Submodule**: メインリポジトリ配下の `services/` 以下は全て Git Submodule であると仮定しています。
