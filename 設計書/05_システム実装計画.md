# 05_システム実装計画書 (System Implementation Plan)

## 1. 前提
- **対象ドキュメント**:
    - `01_要件定義.md`
    - `04_システム設計.md`
- **技術スタック**:
    - **Frontend**: Next.js 15 (App Router), Tailwind CSS
    - **Worker**: Python 3.13, `uv`, Pandas
    - **Auth**: Keycloak (Docker), PostgreSQL
    - **Infrastructure**: Docker Compose (Docker Desktop on Windows / WSL2)

## 2. 実装方針
### 2.1 アーキテクチャパターン
- **Frontend**: Next.js App Router (Server Actions / Server Components 優先)
- **Worker**: Functional Programming 志向 (副作用の分離)。`processor`, `client`, `notifier` に分割。
- **Auth**: Authorization Code Flow with PKCE。Keycloak の `realm-export.json` による構成管理。

### 2.2 ディレクトリ構成・共通処理
- **Python (`uv`)**: `pyproject.toml` による一元管理。`src/` 配下にロジックを配置。
- **共通処理**:
    - **ログ**: Python は `logging` (JSON形式出力)、Next.js は `pino` 等を用いた構造化ログ。
    - **エラー**: 独自の `AppError` クラスを定義し、Discord 通知用のメタデータを保持。

## 8. リスクと対策
- **実装漏れ**: ホワイトリスト照合後の Keycloak 登録画面への「強制遷移」はブラウザバックで回避される可能性がある。
    - **対策**: Middleware レベルでセッション未確立時のリダイレクトを徹底する。
- **技術的難所**: Docker Desktop on Windows におけるファイル権限 (`chmod`) の不整合。
    - **対策**: コンテナ内実行ユーザーを UID/GID で調整するか、WSL2 バックエンドの使用を必須とする。

## 9. CI/CD連携
- **ビルドフロー**: ローカルでの `docker compose build` 検証。
- **テスト実行**: PR 作成時に各エンジニアのローカル環境で `run-tests.sh` を実行し、結果をコメント。
- **デプロイ**: `main` ブランチへのマージ後、手動で ConoHa VPS 上で `git pull` & `up -d` を実施。
