# 03_テスト実装計画 (Test Implementation Plan)

## 1. 前提
- **対象ドキュメント**:
    - `01_要件定義.md` (要件ID: UC_xx, 4.x)
    - `02_テスト設計.md` (テストケースID: TC_xxx)
- **技術スタック**:
    - **Frontend (Next.js)**: Vitest, React Testing Library, Playwright (E2E)
    - **Worker (Python)**: pytest, pandas
    - **Auth**: Keycloak (Docker-based)
    - **環境**: Docker Compose (Local)

## 2. テスト実装方針
### 2.1 レベル別実装アプローチ
- **Unit/Integration**: 外部API（Google/Tikfinity）との境界はモック化し、ロジックの網羅性を高める。
- **E2E**: ローカルの Docker Compose 環境を立ち上げ、Playwright を用いて実ブラウザ操作をシミュレーションする。
- **Keycloak シード管理**: `realm-export.json` を用いた構成管理。テスト実行前にコンテナへインポートし、テストユーザーの状態を固定する（ベストプラクティス）。

### 2.2 命名規則
- **ファイル名**: `[機能名].[test/spec].[ts/py]`
- **関数名**: `test_[要件ID]_[期待挙動]`

## 6. リスクと対策
- **環境依存**: Windows 上の Docker ネットワーク起因で E2E が失敗する可能性がある。
    - **対策**: `localhost` ではなく `host.docker.internal` またはコンテナ名での通信を徹底する。
- **シークレット欠落**: ローカル実行時に `.env` が不足しテストが失敗する。
    - **対策**: `.env.example` を整備し、テスト実行時に不足を検知するバリデーションをスクリプトに入れる。

## 7. 自動化・ローカル運用連携
- **実行タイミング**: 
    - 開発中: 各サブプロジェクトの `watch` モード。
    - PR作成/マージ前: `run-tests.sh` による全件実行。
- **カバレッジ測定**:
    - Frontend: `vitest --coverage`
    - Python: `pytest --cov=src`
- **失敗時のフロー**: 
    - 失敗したテストIDを `02_テスト設計.md` と照らし合わせ、原因を特定。
    - 修正後、再度 `run-tests.sh` をパスするまでマージ不可とする。
