```
あなたは世界クラスのソフトウェアエンジニアリングアシスタントです。

# 目的
本プロジェクトの品質・セキュリティ・運用安定性を担保するための統制事項を定義する。

# 成果物
00_統制事項.md を作成すること。

# 要件
- 抽象論ではなく「具体的なルール」「判断基準」「禁止事項」を含める
- 各項目に以下を含める
  - 目的
  - ルール（必須 / 推奨を区別）
  - 理由
  - 違反時のリスク
  - チェック方法（レビュー観点 or 自動化方法）
- 開発者が迷わないレベルまで具体化する
- 必要に応じてチェックリスト形式も含める

# カバー範囲
最低限以下を含めること
- コーディング規約
- テスト方針
- セキュリティ（認証・認可・データ保護）
- ログ・監視
- エラーハンドリング
- 依存ライブラリ管理
- Git運用（ブランチ戦略・レビュー）
- デプロイ・リリース管理
- パフォーマンス・負荷対策
- 法令・利用規約遵守

# 出力フォーマット
- Markdown形式
- 見出し構造を整理する
- 実務でそのまま使える粒度にする

# 進め方
1. まず不足している情報を質問してください（最大10個まで、重要度順）
2. 私の回答をもとに統制事項を作成してください
3. 不明点は仮定を置き、その旨を明記してください

# 前提（仮）
- 未指定の項目は一般的なWebアプリ開発として仮定してよいが、必ず明記すること
```

```
- 抽象的な表現は禁止（例：「適切に処理する」など）
- 「誰が・いつ・何を・どうする」を明確にする
- 各機能に受け入れ条件（Acceptance Criteria）を必ず記載する
- 正常系・異常系の両方を書く

# 含める内容
最低限以下を網羅すること

- 目的

## 2. ステークホルダー
- 利用者
- 管理者
- 関係者

## 3. ユースケース / ユーザーストーリー
- 役割ごとに整理
- 優先度（Must / Should / Could）

## 4. 機能要件
各機能について以下を記載
- 機能概要
- 入力
- 出力
- 処理内容

- パフォーマンス（例：レスポンス時間）
- 可用性
- セキュリティ
- ログ・監視
- スケーラビリティ
- 法令・利用規約対応

## 6. 画面 / API 要件
- 画面一覧 or API一覧
- 各項目の仕様

## 7. データ要件
- エンティティ一覧
- 主な属性
- 制約

## 8. 業務フロー / ユーザーフロー
- テキストでもよいが具体的に

## 9. 制約・前提条件
- 技術的制約
- 運用制約

## 10. リスクと未確定事項
- 要確認事項を明示
```

```
あなたは世界クラスのQAエンジニアです。
単なるテスト項目ではなく、実装に直結する構造を持った「テスト設計書」を作成してください。

# 目的
テスト観点・テストケースだけでなく、実装しやすいテスト構造（ファイル構成）まで設計する

# 成果物
02_テスト設計書.md を作成すること

# 進め方
1. 不足している情報を質問してください（最大15個、重要度順）
2. 回答をもとにテスト設計書を作成してください
3. 不明点は仮定を置き、明記してください

# 必須要件（重要）
- 正常系・異常系・境界値を必ず含める
- 観点ベースで網羅的に整理する
- 期待結果は具体的に記述する
- テスト構造（論理単位）とファイル構成を紐づける
- ただし実装計画レベルの詳細（細かすぎる関数名など）には踏み込まない

# 含める内容

## 1. テスト方針
- テストレベル（Unit / Integration / E2E）
- テスト対象 / 対象外

## 2. テスト観点一覧
- 入力バリデーション
- 境界値
- 状態遷移
- 権限・認可
- エラーハンドリング
- 外部連携
- セキュリティ
など

## 3. テスト構造設計（重要）
- 機能ごとのテスト単位を定義する
- どの単位でテストを分割するかを明確にする

例：
- ユーザー機能
  - 登録
  - ログイン
  - プロフィール更新

## 4. テストファイル構成（設計レベル）
- ディレクトリ構造（例：tests/unit/, tests/integration/）
- 機能単位でのファイル分割方針

例：
tests/
  unit/
    user/
      user_create.test.ts
      user_login.test.ts
  integration/
    auth_flow.test.ts

※命名規則・粒度のルールを記述する（具体ファイルの網羅までは不要）

## 5. テストケース一覧
各テストケースに以下を含める

- テストID
- 対象機能
- 対応テスト構造（どのファイル/単位に属するか）
- 観点
- 前提条件
- 入力
- 手順
- 期待結果
- 優先度

## 6. 異常系・エッジケース
- 明示的にまとめる

## 7. データ設計
- テストデータパターン
- 境界値・不正値

## 8. カバレッジ戦略
- 何をもって網羅とするか
- 要件との対応

# 出力フォーマット
- Markdown形式
- テストケースは表形式

# NG
- ファイル構成がない
- テスト構造が曖昧
- 実装レベルに踏み込みすぎる（関数単位の詳細など）
- 曖昧な期待結果
```

# 03_テスト実装計画.md
```
あなたは世界クラスのQAエンジニア兼テストアーキテクトです。
単なる方針ではなく、実際に開発チームがそのまま実行できる「テスト実装計画書」を作成してください。

# 目的
テスト設計書をもとに、テストコードを確実に実装・管理・完了できる状態を作る

# 成果物
03_テスト実装計画.md を作成すること

# 進め方
1. 不足している情報を質問してください（最大15個、重要度順）
2. 回答をもとに計画書を作成してください
3. 不明点は仮定を置き、その旨を明記してください

# 必須要件（重要）
- 要件ID ↔ テストケースID ↔ テストコード の対応関係を明確にする
- 実装タスクとして分解する（粒度は1タスク=1〜4時間程度）
- 各タスクに完了条件（Definition of Done）を設定する
- 優先度・依存関係を明示する
- 進捗管理できる構造にする（チェックリスト形式）

# 含める内容

## 1. 前提
- 対象要件定義書
- 対象テスト設計書
- 技術スタック（テストフレームワークなど）

## 2. テスト実装方針
- テストレベルごとの実装方針（Unit / Integration / E2E）
- モック・スタブの方針
- テストデータ管理方針
- 命名規則

## 3. トレーサビリティマトリクス
表形式で以下を整理
- 要件ID
- テストケースID
- テストコード名（ファイル / 関数）
- テスト種別
- 実装ステータス（未着手 / 進行中 / 完了）

## 4. テスト実装タスクリスト
各タスクに以下を含める

- タスクID
- 対応テストケースID
- 内容（何を実装するか）
- 実装方針（どうやってテストするか）
- 完了条件（Definition of Done）
- 優先度（High / Medium / Low）
- 依存関係
- 見積もり（時間）

※チェックリスト形式にすること

## 5. 完了チェックリスト
- 全テストケースに対応するコードが存在する
- テストが再現性を持つ
- テストがCIで実行される
- フレーク（不安定テスト）がない
- 要件カバレッジが100%に達している

## 6. リスクと対策
- 実装漏れリスク
- テストの過不足
- 環境依存問題 など

## 7. 自動化・CI連携
- 実行タイミング（PR時 / 定期）
- カバレッジ測定方法
- 失敗時の対応フロー

# 出力フォーマット
- Markdown形式
- 表とチェックリストを適切に使う

# NG
- 実装できないレベルの抽象的な説明
- トレーサビリティがない
- タスク分解されていない
```
