# 02_テスト設計書 (Test Design)

## 1. テスト方針
### 1.1 テストレベルの定義
- **Unit Test (単体テスト)**: 各サブプロジェクト（Next.js / Python）内部のロジック、関数単位の検証。外部APIはモック化する。
- **Integration Test (結合テスト)**: サブプロジェクト間の連携、および Google Sheets API / Tikfinity API との通信インターフェースの検証。
- **E2E Test (エンドツーエンドテスト)**: `docker-compose` で構築されたローカル環境における、ブラウザを介したユーザーシナリオの検証。

### 1.2 テスト対象・対象外
- **対象**:
  - 認証フロー（スプレッドシート照合 〜 Keycloakサインアップ）
  - データ加工パイプライン（Excel読み込み 〜 スプレッドシート同期）
  - ガチャロジック（ポイント消費 〜 景品抽選）
  - ランキング表示（CSVパース 〜 認可制御）
- **対象外**:
  - Keycloak 内部の認証プロトコル自体の脆弱性テスト（既製品のため）
  - TikTok Live Backstage サイト自体の操作（手動運用の前提）

## 2. テスト観点一覧
| 観点カテゴリ | 主な検証内容 |
| :--- | :--- |
| **認証・認可** | ホワイトリスト（Sheets）にないユーザーの拒否、ログイン済みセッションの維持、他者データへのアクセス制限 |
| **入力値検証** | 不正なユーザー名形式、壊れたExcelファイル、必須パラメータ欠落時のAPI挙動 |
| **データ整合性** | ExcelからSheetsへの変換精度、二重取り込み時の挙動、計算ロジック（スコア等）の正確性 |
| **外部連携** | Tikfinity APIダウン時のフォールバック、Google APIレートリミット時のリトライ（またはエラー通知） |
| **境界値** | 大規模なデータ行数（1,000行以上）でのパース、景品確率 0% / 100% の挙動 |
| **エラーハンドリング** | Discord Webhook通知の成否、エラーログの秘匿化（マスク処理）の確認 |

## 3. テスト構造設計
機能およびレベルに基づき、以下の論理単位でテストを構成する。

### 3.1 認証基盤 (Auth Framework)
- **Whitelist Verification**: スプレッドシート照合ロジック。
- **OIDC Flow**: Keycloakへのリダイレクトおよびトークン取得。

### 3.2 データパイプライン (Data Fetcher)
- **File Ingestion**: 指定ディレクトリからのファイル検知と読み込み。
- **Processing**: Pandasによるデータクレンジングと集計。
- **Synchronization**: Google Sheetsへの書き込み処理。

### 3.3 コンテンツ機能 (Corporate & Ranking)
- **Gacha System**: Tikfinity連携と抽選処理。
- **Ranking Display**: CSVデータの動的なフィルタリングと表示。

## 4. テストファイル構成（設計）
各サブプロジェクトのリポジトリ内に `tests/` ディレクトリを配置し、以下の命名規則で管理する。

```text
services/
  [subproject]/
    tests/
      unit/           # 命名: [機能名].spec.ts / .test.py
      integration/    # 命名: [連携先]_integration.spec.ts
      fixtures/       # テストデータ（dummy.xlsx, mock_user.json 等）
docker-compose/
  tests/
    e2e/              # 命名: [シナリオ名].e2e.spec.ts
      auth_flow.e2e.spec.ts
      data_pipeline.e2e.spec.ts
```

## 6. 異常系・エッジケース詳細
### 6.1 通信・外部サービス障害
- **Google API 403/429 エラー**: 認証エラーやレートリミット発生時に、システムがハングアップせず、適切にエラーログを残して終了すること。
- **Tikfinity API 500 エラー**: ガチャ実行中に外部APIが落ちた場合、ユーザーのポイントが「二重減算」または「未減算で景品付与」されないことを検証する。

### 6.2 データ不備
- **空のExcelファイル**: `/data` に 0バイトのファイルやデータなしのExcelが置かれた場合、スキップ処理と通知が行われること。
- **スプレッドシートの重複行**: ホワイトリストに同一ユーザー名が複数ある場合、最初の1件を採用する等の優先順位ロジックの検証。

## 7. データ設計
### 7.1 モックデータ
- **Sheets Mock**: Google Sheets APIのレスポンスをシミュレートするJSON。
- **TikTok Excel Dummy**: `openpyxl` 等で生成した、様々なケース（全カラムあり、一部欠落、不正な日付形式）を含むテスト用xlsx。

### 7.2 E2E用テストデータ
- **Local Keycloak DB**: テスト実行前に、特定のテストユーザーが登録されていない（または初期状態の）レルムをインポートして使用する。
- **Test Spreadsheet**: 本番とは別の、テスト用SheetIDを使用して検証を行う。

## 8. カバレッジ戦略
### 8.1 網羅性指標
- **Requirement Coverage (100%)**: `01_要件定義.md` に記載された全ての「受け入れ条件」に対応するテストケースが存在すること。
- **Code Coverage (C0 >= 80%)**: 特にビジネスロジックを含む `data-fetcher` と ガチャAPI において、命令網羅 80% 以上を CI で強制する。

### 8.2 判定基準
- 全ての `High` 優先度テストケースがパスしていること。
- E2Eテストにおいて、クリティカルパス（ログイン〜データ閲覧〜ガチャ実行）が失敗しないこと。

---

## 9. 備考（仮定事項）
1. **認証テスト**: Keycloak の UI テストは Playwright 等のヘッドレスブラウザを用いて実施することを前提とする。
2. **連携テスト**: Tikfinity API のテストには、開発者用のサンドボックスAPIキー、またはモックサーバー（Prism 等）を使用することを前提とする。
